## Access Flow

1. The site opens on a black WebGL intro with a white particle field and four staged lines:
   - `How lucky we are to be able to say "I remember".`
   - `Let's share our memories to fight Alzheimer's disease.`
   - `This site will gradually disappear`
   - `if it is not regularly given memories.`
2. After the intro delay, or after clicking the intro, the home state appears.
3. The home state keeps the black particle world, the top-left `I remember,` sprite logo, the top-right memory/fade copy, the circular `13% fade from memories` meter, FRM branding, share controls, sound, donate, terms, credits, and language menu.

## Home Interactions

- Center plus button: enters the add-memory flow.
- Lower center eye button: enters the explore-all-memories flow.
- Footer share buttons: open the original-style Facebook, Twitter, and Google Plus share URLs.
- Sound button: toggles sound for the add confirmation chime.
- Terms and Credits: open right-side black panels with close controls.
- Language menu: preserves the original `LANGUE_EN` affordance and hover menu. English is served from `/en`; French is restored from the archived root page at `/` and `/fr`.

The archived footer script had a dead Credits path: `.footer-link-credits` was selected and `showCredit()` existed, but the click handler was not bound. `public/js/revival-runtime.js` restores that single footer binding while reusing the original hidden `.credit` markup and artwork.

Footer control verification now covers the original delayed language-switch handler, the three share buttons, the donation link interception path, and the sound toggle. The English-to-French path resolves to `/?ln=fr`; the French-to-English path resolves to `/en?ln=en`, matching the archived `BASE_URL + data-url-suffix + ln` logic.

## Explore Interactions

- A left rail appears with a circular map/minimap, search, selected-search label, and add button.
- A tutorial sequence overlays the particle field:
  - `Use your mouse to move`
  - `Scroll or double click to zoom in`
  - `Click to watch a memory`
- The particle canvas responds to mouse movement, wheel zoom, and double-click zoom.
- Memory points are clickable and open a black memory detail panel with matching tags and social sharing.
- Direct memory links follow the archived offset logic: the URL id is the internal post id plus `POST_ID_OFFSET = 1248`. For example, `/en/memory/4793` opens archived post `3545` (`CLCLC`, `today with you in HK`).
- The restored detail panel preserves the original close/share/tag behavior: share buttons open the original Facebook, Twitter, and Google Plus URL shapes; close returns the English route to `/en/`; related message tags show the matched-count hover copy and can launch the filtered search state.

## Search Logic

- Search opens a centered oversized italic input with a bottom rule and search icon.
- Input normalizes spaces and punctuation toward the original tag-route behavior.
- The historical backend endpoint is no longer available through the archive, so Vite serves local shims for `api/search-posts`, `api/related-post-count`, and `api/auto-complete-tags`.
- The April 2014 Wayback CDX index has no available `api/search-posts` capture for this page. A later 2014 API capture exists, but it contains newer July data and does not match the April 30 visual state.
- The shim preserves the original JSON/JSONP response shapes expected by the archived `postController`.
- The shim now selects the archived English or French default-memory dataset from `ln=en` or `ln=fr`; the default datasets are extracted from the restored HTML `DEFAULT_POSTS` payloads at startup instead of committed `data/` files.
- The API datasets now merge additional `DEFAULT_POSTS` recovered from clean Wayback page snapshots at or before the requested April 30 capture. This expands the target-date backend surface to 706 English posts and 489 French posts while keeping the initial HTML fixtures unchanged at the original 200-post target snapshot.
- French search now folds accents for matching while keeping the original French data and UI text intact. Queries such as `été`, `fête`, and `école` resolve to the same word-boundary matches as their unaccented route tags instead of collapsing to one-letter fragments.
- Matching queries highlight filtered memory points and add the selected search tag to the left rail.
- Autocomplete preserves the archived interaction: typing at least three characters calls `api/auto-complete-tags/:fragment`, shows inline hint chips, clicking a hint submits that completed tag, and no-result searches keep the search layer open with the original interpolated not-found copy.

## Language And Legal Routes

- `/en` and `/en/memory/:publicId` serve the archived English page and English memory dataset.
- `/`, `/fr`, `/memory/:publicId`, and `/fr/memory/:publicId` serve the archived French page and French memory dataset.
- Direct memory ids still use the original `POST_ID_OFFSET = 1248`; for example `/memory/5025` opens French post `3777` (`Rozenn`), while `/en/memory/4793` opens English post `3545` (`CLCLC`).
- `/legal`, `/fr/legal`, and `/en/legal` serve the standalone archived French legal page with the recovered `css/legal.css`.
- `/close.html` serves the archived social-share callback page, whose only runtime behavior is `window.close();`.
- `/api/instagram-token-callback` serves the archived Instagram OAuth callback shape: it reads `access_token` from `window.location.href`, calls `window.opener.captureInstagramToken(token)`, closes the popup, and falls back to `/` if closing fails.
- The current Wayback French root capture includes French HTML and `LANG = 'fr'`, but its inline `redirectTo` now points to the archived English route. The revival keeps the archived French content reachable at `/` and `/fr` by suppressing that stale redirect.

## Add-Memory Logic

- Upload step: circular particle/upload zone with Facebook, Instagram, and local upload controls.
- Social album choices still use the archived Facebook/Instagram UI code paths, with local JSONP compatibility payloads replacing the unavailable 2014 third-party APIs.
- Facebook now opens the original album picker, opens the mock album, selects a photo, and reaches the original adjustment step. The original adjustment animation first shows the crop ring, then settles into the Back/Confirm controls after the archived timing completes.
- Instagram now opens the original photo picker, selects a photo, and reaches the original adjustment step without an external OAuth popup.
- Local image upload posts to the local `api/upload-image` shim and then loads the original-style `uploads/tmp/:fileId/resized.jpg` preview path.
- URL-based social image upload requests now use the submitted same-site image URL when present, matching the original `api/upload-image` contract used by the Facebook/Instagram picker paths instead of falling back directly to the static `revival-upload` id.
- Invalid local image dimensions are stopped before upload with the archived error blocker copy, then dismiss back to the upload-method choices without calling `api/upload-image` or entering adjustment.
- Uploaded image bytes are retained in the current Vite server session and are replayed through both the original temporary and posted-image URL shapes: `uploads/tmp/:fileId/resized.jpg`, `uploads/posts/:fileId/resized.jpg`, and `uploads/posts/:fileId/thumb.jpg`.
- When no binary file is present, the static `revival-upload` image remains as a compatibility fallback for API-only checks.
- The `api/post` shim accepts the original form fields and returns the submitted name, message, image offset, `fileId`, and 600 x 600 resized dimensions in the original response envelope.
- Adjustment step: framed image preview, back, and confirm.
- Message step: name, email, memory text, terms checkbox, terms link, back, and disabled confirm until valid.
- The adjustment Back control returns to upload methods; the message Terms link opens the original Terms panel over the add flow; closing Terms returns to the filled message step; the terms checkbox enables and disables Confirm; message Back returns to adjustment.
- Confirmation step posts to the local `api/post` shim, shows the original `Thank you` state, and plays `add.mp3` when sound is enabled.
- The `Thank you` share buttons preserve the archived `POST_ID_OFFSET` logic: Facebook, Twitter, and Google Plus target `/en/memory/:publicId` for the newly submitted post, with Facebook returning to `/close.html`.
- Submitted memories are persisted to `.revival-storage/submitted-posts.json`, and uploaded image bytes are persisted under `.revival-storage/uploads/`. The folder is ignored by git, but a restarted local Vite server can still resolve the submitted memory's original-style public URL and posted image paths.

The archived runtime selects `audio/add.mp3` or `audio/add.ogg`, and `audio/loop.mp3` or `audio/loop.ogg`, through the original `Modernizr.audio.mp3 ? "mp3" : "ogg"` branch. Wayback CDX contains archived MP3 captures for both sounds but no OGG captures. The local OGG files are therefore derived from the recovered MP3 files with Vorbis encoding so the original fallback branch remains functional in non-MP3 audio environments. This is verified by `qa/current-audio-format-fallback-contract-audit.json`.

The current Wayback capture observes the original `uploads/posts/*/thumb.jpg` requests, but many timestamp-specific asset fetches are not reliably available from the archive. The local Vite middleware now serves the `revival-upload` image as a compatibility fallback for archived default post `thumb.jpg` and `resized.jpg` requests so the restored scene and social/photo flows do not accumulate 404s.

A CDX-wide post-image intersection check found 5038 archived `uploads/posts/*` image ids in Wayback, but none of the initial 400 image ids referenced by the April 30 English/French default-memory HTML fixtures matched. The evidence is stored in `qa/default-image-cdx-intersection.json`. This makes the fallback a documented archive limitation, not a missing local lookup.

A later prefix CDX recovery found 199 real archived `thumb.jpg` files whose image ids intersect the expanded April 30-or-earlier target-date datasets. Those recovered thumbnails are stored under `public/uploads/posts/<img>/thumb.jpg`, copied into `dist/uploads/posts/<img>/thumb.jpg`, and served by Vite before the compatibility fallback. The recovery and HTTP routing checks are stored in `qa/current-recovered-post-image-assets-audit.json` and `qa/current-recovered-post-image-routing-audit.json`. No matching `resized.jpg` files were available from the archived intersections, so full-size archived post images still use the documented fallback when a recovered upload is not present.

Additional page-snapshot recovery found 1379 unique English posts and 857 unique French posts across April-May captures. The revival intentionally includes only snapshots at or before the requested April 30 source moment, yielding 706 English and 489 French target-date posts; later May posts are documented but excluded to avoid introducing future content into the April 30 reconstruction. The source audit is `qa/current-recovered-post-snapshot-source-audit.json`.

The local `img/colorMap.png` response is delayed by about 1000 ms to keep the first-run preloader cadence close to the current Wayback capture without fully copying Wayback network jitter. The source capture varied between runs after cache warmup, so this is a conservative timing shim rather than a hard clock lock.

The revival runtime also forces the archived app onto a 1x WebGL canvas backing store before `main.js` initializes. On the current Chrome/Wayback source page, the visible 1512 x 745 canvas also has a 1512 x 745 backing store even on a DPR 2 display; without this shim, the local canvas initialized at 3024 x 1490 and rendered the additive particle field too bright and dense.

The revival runtime catches rejected `HTMLMediaElement.play()` promises from modern autoplay policy. The original sound controls and audio requests are left intact; this only prevents rejected autoplay from becoming an unhandled JavaScript error during automated checks.

The revival runtime now also intercepts stale third-party script insertion for Google Analytics and the archived Facebook SDK. The original `_gaq` queue shape is preserved, and the Facebook code path is fulfilled by the local `FB` mock used by the archived album/photo picker. This keeps the revival self-contained while preserving the original UI flow.

Residual Wayback-wrapped social URL literals recovered from the archived `main.js` download were restored to their original targets for Twitter mention/hash links, Facebook Graph/SDK URLs, and Instagram OAuth/API URLs. The local runtime still intercepts the Facebook/Instagram endpoints needed for self-contained revival behavior.

The Vite shim also returns a lightweight `204` for `/favicon.ico` so browser-default favicon probing does not create a local-only 404 console error. The original page markup is otherwise left unchanged.

QA artifacts are kept under the repository-level `qa/` folder only. Duplicated `public/qa/` screenshots and the old duplicated `public/assets/` tree were removed so production builds no longer expose test screenshots or legacy asset aliases as site assets. The Vite shim also returns `404` for `/qa/`, `/assets/`, and local project-internal files during dev so those paths do not fall through to the app shell.

## Framework Choice

The revival uses Vite as a static host and API shim around the original site files. React was removed after comparison showed the 2D recreation could not reach 1:1 fidelity. The runtime path is now the archived HTML/CSS/JS and original WebGL/Three interaction logic.

For an actual production-style run, `npm start` launches `server.mjs`, a small Node HTTP server that reuses the same revival middleware as the Vite dev and preview paths, then serves the built `dist/` assets with byte-range support for audio. This keeps `/en`, `/fr`, `/legal`, direct memory URLs, search/autocomplete/related-count APIs, upload/post submission, restored public uploads, favicon handling, and project-artifact isolation available outside Vite preview.
